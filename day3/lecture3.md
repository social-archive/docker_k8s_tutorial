# [Day 3] 이론 강의: 버전관리와 이미지 태그 반영

> 💡 **쉽게 이해하는 비유 (Analogy Box)**
> - **'최신제품' 스티커 덮어쓰기 vs 영구 시리얼 넘버 각인**
>   - `latest` 태그 배포는 공장에서 매일 새로 생산되는 제품 상자마다 기존 구형 상자에서 떼어낸 '최신제품' 스티커(latest)를 다시 뺏어 붙여 창고에 보관하는 것과 같습니다.
>   - 어느 날 출고된 제품에 심각한 결함이 발생해(버그 발생) 반품하거나 제조 라인을 역추적하려 해도, 상자에 시리얼 넘버가 없어서 이것이 어제 오후에 조립된 것인지 오늘 아침에 조립된 것인지 구별할 수 있는 방법이 전혀 없습니다.
>   - **short SHA 태그**는 제품을 만들 때마다 당시 설계된 기안서 번호(**7자리 깃 커밋 번호**)를 지워지지 않는 **낙인(시리얼 넘버)**으로 제품 본체에 영구 각인하는 것입니다. 설계도가 1줄이라도 바뀌면 제품 번호도 완전히 달라지므로, 언제 만든 어떤 제품인지 단 1초 만에 배포 장부(Git 로그)와 대조해 이력을 추적할 수 있습니다.

---

## 1. 없으면 어떤 점이 불편한가?

개발 중인 애플리케이션의 버전을 관리할 때 태그를 항상 `latest`라는 고정 포인터 문자열(`todo-app:latest`)로 덮어씌워 배포 스택을 가동하면, 실무 운영 및 장애 복구 단계에서 다음과 같은 파멸적인 재앙을 겪게 됩니다.

* **이미지 다운로드 무시 현상과 버전 미반영 에러 (`imagePullPolicy` 문제)**
  - 쿠버네티스의 기본 이미지 다운로드 정책은 **`imagePullPolicy: IfNotPresent`** 입니다. 이는 해당 노드 장비에 대상 태그의 이미지가 이미 다운로드되어 있다면 레지스트리에서 다시 받아오지 않는 정책입니다.
  - 소스 코드를 고쳐 `latest` 이미지를 레지스트리에 새로 밀어 넣었더라도, 워커 노드 로컬 캐시에 예전 `latest` 이미지가 존재하면 쿠버네티스는 새 이미지를 다운로드받지 않고 구형 이미지를 그대로 띄워버려 배포 누락 현상이 일어납니다.
* **롤백(Rollback) 기능의 원천적 불가능화**
  - 신버전 배포 직후 데이터베이스 커넥션이 폭발하는 치명적인 에러를 발견했습니다. 10분 전 정상 기동 상태로 클러스터를 원복하기 위해 `kubectl rollout undo` 명령을 수행합니다.
  - 하지만 이미지 레지스트리에는 `latest` 바이너리 단 하나만 덮어쓰여 존재할 뿐, 과거 1.0 이미지 스냅샷이 흔적도 없이 지워졌기 때문에 이전 상태로 되돌아가는 롤백 복구가 아예 불가능합니다.
* **노드 간의 기상천외한 소스 코드 버전 파편화**
  - 클러스터의 여러 워커 노드 장비 중 노드 A는 `latest`를 어제 아침에 풀(Pull) 받았고, 노드 B는 오늘 새벽에 풀 받았습니다. 
  - 동일한 `latest` 태그이지만 노드 A와 B에서 돌고 있는 소스 코드 버전이 달라져, 사용자가 접속하는 서버 위치에 따라 간헐적으로 버그가 터지는 기괴하고 추적 불가능한 미스터리 버그가 발생합니다.

---

## 2. 왜 필요할까?

`latest`라는 가변 포인터 태그는 **어떤 특정 시점의 이미지 상태(Immutable State)를 고유 지시해 주는 영구적인 지표가 될 수 없기** 때문입니다.

인프라의 완벽한 재현성과 배포의 안정성을 확보하려면 다음과 같은 규칙이 준수되어야 합니다.
1. **이미지의 불변성 (Immutability)**: 한 번 빌드되어 레지스트리에 입고된 이미지는 결코 변경되거나 덮어쓰여질 수 없으며, 모든 빌드에는 고유한 시리얼 넘버 태그가 부착되어야 합니다.
2. **배포 매크로 추적성**: 깃 저장소의 배포 설계도(YAML) 텍스트 파일 내에 명시된 이미지 태그 문자열이 변경됨과 동시에 클러스터 상태가 동기화되어야 하며, `git log`를 조회하는 행위만으로 실시간 인프라 가동 형상이 증명될 수 있어야 합니다.

## 3. 이것은 무엇인가?

> **핵심 한 줄 요약**:
> *"고유 이미지 태깅은 **빌드마다 7자리의 깃 번호표(short SHA)를 지울 수 없는 고유 낙인**으로 이미지에 새기고, 배포 파일에 명문화하여 **실행 중인 소프트웨어 버전을 100% 추적 가능하게 만드는 버전 가시성 기술**이다."*

<details>
<summary><b>🔍 7자리의 깃 번호표: Git Short SHA 태깅의 유일무이성</b></summary>

깃(Git) 버전 관리 도구는 개발자가 커밋(Commit)을 생성할 때마다 변경된 코드 내용, 작성자, 시간 정보를 조합하여 40자리의 고유한 암호학적 해시값(SHA-1)을 생성합니다.
- 이 중 앞의 **7자리 문자열 (Short SHA)**을 잘라내어 도커 이미지 태그(예: `ghcr.io/username/todo-app:a1b2c3d`)로 삼는 전략입니다.
- 이 7자리 조합은 한 저장소 내에서 겹칠 확률이 천만분의 일 수준이므로, 사실상 전 세계에서 유일무이한 고유 이미지 식별 시리얼 번호표 역할을 안전하게 수행합니다.
</details>

<details>
<summary><b>🔍 가장 완벽한 불변 포인터: Image Digest (SHA256 체크섬)</b></summary>

Short SHA 태그도 훌륭하지만, 해커가 깃 저장소를 임의 조작하여 동일한 태그로 악성 이미지를 덮어씌울 가능성이 0.0001%라도 존재합니다.
- **Image Digest (콘텐츠 주소 지정)**: 이미지의 레이어 데이터 전체를 암호학적으로 해시한 SHA256 체크섬 고유값입니다. (예: `ghcr.io/user/app@sha256:77a83d4c...`)
- 이 Digest 포인터는 단 1비트의 코드 조작도 허용하지 않는 **가장 엄격한 완벽 불변성**을 제공하므로, 대기업의 상용 금융 서비스 등 철저한 보안 규격 배포망에서는 Short SHA보다 이 Digest 주소를 직접 매핑하여 배포하는 사상을 지향합니다.
</details>

<details>
<summary><b>🔍 인간의 개입을 배제하는 자동화: GitOps Auto-Update 파이프라인 사상</b></summary>

매번 빌드가 끝날 때마다 개발자가 깃허브 Actions 로그를 열어 7자리 Short SHA 코드를 복사하고, 배포용 `values.yaml` 파일의 `image.tag` 라인을 직접 수정해 푸시하는 것은 번거로운 휴먼 에러의 근원이 됩니다.
- **GitOps Auto-Update (자동화)**:
  - CI 빌드가 성공하여 새 이미지가 사출되면, GitHub Actions Runner가 즉시 배포 매니페스트 저장소를 Checkout 받습니다.
  - Runner 내부에서 `yq` 또는 `kustomize` 같은 텍스트 조작 도구를 이용해 `values.yaml` 내의 `tag: "old-version"` 문자열을 새롭게 빌드된 `tag: "new-sha"`로 자동 치환합니다.
  - 이후 Actions가 스스로 배포용 저장소에 `Commit & Push`를 수행하여, 사람이 복사 붙여넣기 조작을 전혀 개입시키지 않는 **완전 무인식 지속적 배포(No-touch CD)**를 완성합니다.
</details>

### 📊 latest 태그 덮어쓰기 문제점 vs Short SHA 고유 보관 및 추적 아키텍처

```
[latest 태그 덮어쓰기 방식 - 롤백 불가]
─────────────────────────────────────────────────────────────
Registry : ┌──────────────────────────────────────┐ (덮어씀)
           │ [latest] Binary v1.0 ➡️ Binary v2.0  │ 
           └──────────────────────────────────────┘
Cluster  : Pod1(v1.0 캐시) ──► 재시작 ──► 여전히 v1.0 기동 (배포 누락)
           Pod2(캐시 없음) ──► 재시작 ──► v2.0 기동 (노드 간 소스 버전 파편화)
           ※ 에러 인지 후 롤백 수행 시도 ──► 이전 v1.0 바이너리가 삭제되어 복구 실패!


[Short SHA 고유 태깅 방식 - 완벽한 이식성과 롤백 보장]
─────────────────────────────────────────────────────────────
Registry : ┌──────────────────┐  ┌──────────────────┐
           │ [tag: a1b2c3d]   │  │ [tag: 9f8e7d6]   │ (각각 고유 스냅샷 보존)
           │ Binary v1.0      │  │ Binary v2.0      │
           └──────────────────┘  └──────────────────┘
Cluster  : values.yaml 수정 ──► tag: "9f8e7d6" 커밋 푸시
           Argo CD가 텍스트 변경을 감지하고, 모든 노드에 v2.0 강제 다운로드 배포.
           ※ 에러 인지 후 롤백 수행 ──► values.yaml을 tag: "a1b2c3d"로 되돌려 커밋
           Argo CD가 즉각 인지하여 Registry에서 보존 중인 v1.0 스냅샷을 1초 만에 복구.
```

## 4. 장점과 단점

### 1) 장점
* **인프라 배포의 투명한 감사(Audit) 실현**
  - `git log --oneline` 명령을 통해 어떤 자바 소스 커밋 형상이 실 운영 인프라에 어떤 시점에 적용되었는지 100% 명확한 가시성과 이력 증적을 확보합니다.
* **Argo CD 변경 감지의 즉각성**
  - 배포 파일 내부의 태그 텍스트 값이 물리적으로 변함에 따라 Argo CD가 아무런 딜레이 없이 변경을 인지하고 즉각적인 롤링 배포를 조율하기 시작합니다.

### 2) 단점과 극복 비용
* **수동 복사 작업의 공수 (기본 실습의 한계)**
  - 본 실습의 기본 과정은 동작 이해를 위해 사람이 수동으로 short SHA를 복사해 반영하는 방식을 취하므로, 최초 세팅 시 빌드 로그를 직접 관찰하여 번호를 따와야 하는 수작업 단계가 일부 동반됩니다.

## 5. 어떻게 쓰는가?

빌드 성공 후 산출된 7자리 short SHA 코드를 확인하고, 이를 배포 전용 설정 파일에 수동 동기화하여 버전 이력을 관리하는 실무 제어 절차입니다.

### 1) 실무형 `values.yaml` 버전 매핑 예시
```yaml
# day3/k8s/helm/values.yaml
replicaCount: 3

image:
  repository: ghcr.io/social-archive/todo-app
  # latest를 사용하지 않고, GitHub Actions 실행이 완료된 빌드의 7자리 Short SHA를 입력
  tag: "a1b2c3d"
  pullPolicy: IfNotPresent
```

### 2) 배포 명세서 갱신 및 이력 추적 명령어
```powershell
# 1. 수정한 values.yaml 파일의 이미지 태그 변경 사항을 Git 스테이징 및 커밋 서명
git add day3/k8s/helm/values.yaml
git commit -m "deploy: update app image version to a1b2c3d"

# 2. 저장소에 Push하여 Argo CD가 읽어갈 수 있도록 최종 등재
git push origin main

# 3. 배포 장부의 변천사를 확인하여 과거 어떤 커밋 버전들이 서버에 적재되었었는지 로그 추적
git log --oneline day3/k8s/helm/values.yaml
```
