# [Day 2] 이론 강의: Kubernetes 기본 구조

> 💡 **쉽게 이해하는 비유 (Analogy Box)**
> - **선풍기 수동 버튼 vs 에어컨 자동 온도 조절기**
>   - 수동 명령 방식(Docker CLI)은 선풍기 바람 세기를 사람이 직접 미풍, 강풍 버튼을 눌러가며 더우면 올리고 추우면 끄는 것과 같습니다. 실내 온도를 일정하게 유지하려면 사람이 24시간 내내 버튼 앞에 붙어 감시해야 합니다. 만약 버튼을 누르는 순간 선풍기 전선이 타버린다면(중간 단계 실패), 방 온도를 낮추기 위한 모든 작업은 중단됩니다.
>   - **Kubernetes 선언적 API**는 에어컨 온도 조절기와 같습니다. 사람이 "희망 온도를 24도"로 한 번 맞춰놓으면(선언), 기계 내부의 **온도 조율 센서(조정 루프)**가 방 온도가 24도보다 올라가면 알아서 컴프레셔를 세게 틀고, 추워지면 끄는 과정을 반복하여 사람이 개입하지 않아도 24도를 유지해 줍니다. 
>   - 외부 충격으로 창문이 잠시 열려 온도가 급상승(장애 발생)하더라도, 조절기가 스스로 감지하여 24도에 도달할 때까지 냉방 시스템을 가동합니다.

## 1. 없으면 어떤 점이 불편한가?

기존 Docker나 Docker Compose 환경에서는 시스템의 상태를 수정하거나 서비스 리소스를 확장(Scale-out)할 때 다음과 같은 불편함과 운영적 한계를 겪게 됩니다.

* **명령형 인프라 제어와 '상태 표류(Configuration Drift)' 현상**
  - 서비스 트래픽이 몰려 웹 서버 컨테이너 복제본을 5개로 늘려야 할 때, 담당자가 직접 터미널을 열고 `docker run` 명령어를 포트 번호를 다르게 지정해가며 5번 실행해야 합니다.
  - 이 과정에서 4번째 컨테이너 빌드 도중 디스크 용량이 꽉 차서 실패하면, 인프라는 목표한 5개가 아닌 3개만 실행된 '애매하고 불안정한 중간 상태'로 방치됩니다. 
  - 이처럼 사람이 직접 실행 명령어를 차례로 쳐서 상태를 조작하는 방식을 **명령형(Imperative) 제어**라고 합니다. 명령형 구조는 중간에 단계가 꼬이면 전체 시스템의 실제 상태가 설계 장부와 다르게 표류(Drift)하는 문제를 야기합니다.
* **자가 치유 기능의 전무함과 좀비 프로세스 방치**
  - 백그라운드로 띄워놓은 컨테이너 프로세스가 내부 메모리 릭으로 인해 예기치 않게 죽어버렸습니다. Docker Engine 자체는 컨테이너가 죽었음을 인지하더라도, 이를 스스로 다시 살려내지 않습니다.
  - 결국 개발팀이 직접 쉘 스크립트나 크론탭(Crontab)을 짜서 주기적으로 포트를 찔러보고, 죽어있으면 `docker restart` 명령을 던지도록 땜질식 방어 코드를 작성해야 합니다.

## 2. 왜 필요할까?

Docker CLI는 "이 컨테이너를 실행하라"는 **단발성 명령형 명령어만 수행할 뿐**, 인프라의 전체 형상이 유지되고 있는지를 지속해서 계측하고 복원해 주는 **동적 상태 엔진이 결여되어 있기 때문**입니다.

인프라가 커지고 안정적으로 유지되려면 다음과 같은 아키텍처적 기반이 잡혀야 합니다.
1. **선언형 API (Declarative API)**: "어떻게 도달할지(How)"가 아닌, "원하는 인프라의 최종 모습(What)"을 명확히 선언하고, 이 선언문(YAML 매니페스트)을 중앙 데이터베이스에 영구 등록하여 단일 진실 공급원(SSOT)으로 삼아야 합니다.
2. **지속적 조정 루프 (Continuous Reconciliation Loop)**: 등록된 설계 장부의 값과 실시간 하드웨어에 올라가 있는 컨테이너 상태를 매초 대조 및 감시하며, 차이가 나는 순간 즉각 교정 행동을 실행하는 지능형 피드백 제어 루프가 백그라운드에서 상시 회전해야 합니다.

## 3. 이것은 무엇인가?

> **핵심 한 줄 요약**:
> *"Kubernetes API와 조정 루프는 **원하는 인프라의 최종 형상(설계도)만 제출해두면**, 시스템이 24시간 감시하며 **실제 물리 서버를 설계도와 100% 동일하게 일치시켜 주는 상태 자동 동기화 엔진**이다."*

<details>
<summary><b>🔍 명령형(Imperative) vs 선언형(Declarative) 아키텍처 심층 비교</b></summary>

* **명령형 (How - 어떻게에 집중)**:
  - 예시: "우분투 서버에 접속해라 ➡️ 자바를 설치해라 ➡️ jar 파일을 복사해라 ➡️ 8080 포트로 띄워라."
  - 단점: 실행 중간에 네트워크 에러 등으로 3단계가 실패하면 인프라가 반만 생성된 오염된 상태로 고립됩니다. 동일 명령어를 한 번 더 치면 "이미 존재합니다" 에러를 내며 뻗어버리므로 멱등성(Idempotency)이 깨집니다.
* **선언형 (What - 무엇에 집중)**:
  - 예시: "이 스프링 앱 이미지가 8080 포트로 3개 구동되고 있는 상태를 유지해라."
  - 장점: 인프라의 최종 상태만 정의합니다. 동일한 선언형 YAML 파일을 API 서버에 100번 전달(apply)하더라도, 시스템은 이미 해당 상태가 유지되고 있음을 파악하고 아무런 오버헤드 없이 무반응으로 통과합니다(완벽한 멱등성 보장).
</details>

<details>
<summary><b>🔍 etcd Watch 메커니즘을 통한 이벤트 기반 비동기 반응 조율</b></summary>

쿠버네티스의 모든 리소스 상태는 중앙 저장소인 `etcd`에 저장됩니다.
- API Server는 etcd의 특정 키(예: `/registry/deployments/todo-app`) 값이 변경되면 이를 구독(Subscribe)하고 있는 연관 컨트롤러에게 **비동기 이벤트(Watch Event)**를 즉시 발송합니다.
- 만약 개발자가 `kubectl apply`로 레플리카 개수를 3개에서 5개로 늘리면:
  1. API Server가 etcd 값을 5로 업데이트합니다.
  2. etcd가 Watch 이벤트를 API Server로 쏩니다.
  3. API Server는 이 이벤트를 대기 중이던 `Deployment Controller`에게 중계합니다.
  4. 컨트롤러는 매초 폴링하며 자원을 소모하지 않고도, 값이 변하는 순간 즉각 반응하여 신규 파드를 2개 띄우는 조정 작업을 수행합니다.
</details>

<details>
<summary><b>🔍 조정 루프(Reconciliation Loop)의 3단계 피드백 원리</b></summary>

쿠버네티스 컨트롤러 매니저(`kube-controller-manager`) 내부에는 수십 개의 마이크로 컨트롤러 스레드들이 각각 독립적으로 아래 루프를 끊임없이 돌며 자신의 담당 리소스만 감시합니다.
1. **관찰 (Observe)**: 실제 구동 중인 워커 노드들의 상태를 조회하여 "현재 가동 중인 실시간 파드 수가 2개"임을 알아냅니다.
2. **대조 (Diff)**: 사용자가 YAML로 선언해 둔 목표값 "3개"와 현재 관측값 "2개"를 대조하여 `-1`개라는 격차(Delta)를 도출합니다.
3. **조정 (Act)**: 스케줄러와 Kubelet에 지시를 내려 비어있는 자리에 파드 1개를 새로 생성해 현실을 설계도와 일치시킵니다.
</details>

<details>
<summary><b>🔍 논리적 울타리: 네임스페이스 (Namespace) 격리 사상</b></summary>

쿠버네티스 클러스터를 물리적으로 여러 개 구매하는 것은 비용 낭비가 큽니다.
- **네임스페이스**: 단일 물리 클러스터 자원을 논리적으로 쪼개어 쓰는 가상 클러스터 구획 기술입니다.
- 네임스페이스가 다르면 동일한 이름의 Pod나 Service 리소스를 충돌 없이 생성할 수 있습니다. 
- **주의점**: 네임스페이스는 API 상의 논리적 스코프 격리일 뿐, 기본적으로 네트워크 통신을 물리적으로 완전 차단하지는 않습니다. (완벽한 멀티테넌시 보안 격리를 위해서는 `NetworkPolicy` 리소스를 통해 파드 간의 트래픽 흐름을 통제해야 합니다.)
</details>

### 📊 K8s 선언형 아키텍처와 etcd Watch 기반 조정 루프 흐름

## 4. 장점과 단점

### 1) 장점
* **자율 운영 인프라 달성 (Zero Ops에 한 발짝)**
  - 엔지니어가 개입하지 않아도 물리 서버 장애나 앱 크래시 시 자가 치유(Self-healing)에 의해 단 몇 초 만에 다른 노드에서 컨테이너가 부활하여 장애 시간을 극적으로 단축합니다.
* **인프라 멱등성 확보**
  - 동일한 배포 명세서(YAML)를 클러스터에 몇 번을 던지더라도 항상 선언된 형상을 유지하므로 배포 사고율이 제로에 가까워집니다.

### 2) 단점과 해결책 (k9s 대시보드의 당위성)
* **내부 가시성(Visibility)의 상실**
  - 쿠버네티스는 스스로 파드를 지우고, 다른 노드로 이삿짐을 나르고, IP를 바꾸는 등 끊임없이 자율 조율을 수행합니다.
  - 이로 인해 개발자가 단순 터미널에서 `kubectl get pods` 명령어만 번갈아 쳐서는 실시간 상태 변화나 에러 로그의 흐름을 직관적으로 따라가기 매우 고달픕니다.
* **k9s 도입**: 터미널 기반의 고성능 실시간 모니터링 도구인 **k9s**를 실행하면, 클러스터 내부 리소스들의 상태 전환, 로그 흐름, 이벤트 에러 메시지를 UI 형식으로 실시간 모니터링하며 자율 인프라의 블랙박스 문제를 완전히 해소할 수 있습니다.

## 5. 어떻게 쓰는가?

원하는 상태를 선언한 YAML 매니페스트 파일을 작성하고 이를 클러스터에 반영 및 모니터링하는 기본 명령어 가이드입니다.

### 1) 실무형 `namespace.yaml` 선언서 예시
```yaml
apiVersion: v1
kind: Namespace
metadata:
  name: todo-app
  labels:
    env: development
    project: todo-service
```

### 2) 클러스터 형상 제어 및 모니터링 명령어
```powershell
# 1. 작성한 선언서(YAML)를 적용하여 클러스터 상태를 선언형으로 업데이트
# (만약 이미 namespace가 존재하고 속성이 같다면 아무 변경도 가하지 않습니다)
kubectl apply -f namespace.yaml

# 2. 현재 클러스터에 등재된 논리적 구역(Namespace) 목록 및 활성 상태 조회
kubectl get namespace

# 3. 특정 네임스페이스(todo-app) 내부에 존재하는 모든 API 리소스 일괄 조회
kubectl get all -n todo-app

# 4. 실시간으로 변하는 클러스터의 내부 자율 동작 상태를 가시화해주는 k9s 대시보드 기동
# (k9s 터미널 창 내에서 네임스페이스 이동 및 파드 로그 실시간 추적이 가능합니다)
k9s
```
