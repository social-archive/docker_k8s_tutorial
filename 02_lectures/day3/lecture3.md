# [Day 3] 3-3. 버전관리와 이미지 태그 반영

---

## 오늘 배울 내용
- **주제**: Docker 이미지 태깅 전략, `latest` 태그의 위험성, Git Short SHA 기반 버전 제어 및 배포 추적성 확보
- **목표**:
  - `latest` 태그 사용 시 유발되는 배포 누락 및 버전 파편화 문제 이해
  - 이미지 불변성(Immutability)을 위한 고유 태깅의 중요성 파악
  - Git 7자리 Short SHA를 이미지 식별자(시리얼 넘버)로 사용하는 방법 습득
  - YAML 배포 설정 내 이미지 태그를 갱신하고 Git으로 변경 이력을 추적하는 명령어 실습

---

## 💡 쉽게 이해하는 비유 (Analogy)
- **'최신제품' 스티커 덮어쓰기 vs 영구 시리얼 넘버 각인**
  - **`latest` 태그**: 매일 새로 생산되는 제품 상자에서 기존의 '최신제품' 스티커를 떼어내어 새 상자에 다시 덮어붙이는 것. 제품에 결함(버그)이 나도 스티커 외엔 정보가 없어 언제 만든 불량품인지 알 방법이 없습니다.
  - **Short SHA 태그**: 제품을 만들 때마다 당시의 설계도 번호(7자리 깃 커밋 번호)를 지워지지 않는 낙인(시리얼 넘버)으로 상자에 새기는 것. 1줄이라도 변경되면 번호가 완전히 달라지므로 배포 장부(Git 로그)와 대조해 즉각 추적 복구할 수 있습니다.

---

## 1. latest 태그 사용의 문제점 (1) 배포 누락
- **이미지 다운로드 무시 현상과 버전 미반영**
  - K8s의 기본 이미지 다운로드 정책은 `imagePullPolicy: IfNotPresent` (노드 로컬 캐시에 해당 태그 이미지가 있으면 다운을 스킵함).
  - 소스코드를 수정해 `latest` 이미지를 레지스트리에 새로 덮어썼더라도, 노드가 기존에 받아서 보관 중이던 예전 `latest` 이미지를 그대로 사용해 파드를 띄움으로써 배포가 누락되는 현상이 발생함.

---

## 1. latest 태그 사용의 문제점 (2) 롤백 불가
- **과거 시점으로의 롤백 복구 실패**
  - 신규 버전 배포 직후 중대한 에러를 인지하여, 10분 전 정상 기동 상태로 클러스터를 돌리기 위해 `kubectl rollout undo` 명령어를 던짐.
  - 하지만 레지스트리에는 `latest` 바이너리 단 하나만 계속 덮어쓰여 존재할 뿐, 과거의 1.0 이미지 스냅샷이 이미 사라졌기 때문에 이전 버전 복구가 완전히 불가능함.

---

## 1. latest 태그 사용의 문제점 (3) 파편화
- **노드 간 소스 코드 버전의 은밀한 파편화**
  - 여러 워커 노드 중 노드 A는 `latest`를 어제 다운받았고, 노드 B는 오늘 새벽에 다운받음.
  - 동일한 `latest` 태그 명칭이지만 노드 A와 B가 실행하는 실제 바이너리 버전이 달라져, 사용자가 접속하는 서버 위치에 따라 간헐적으로 버그가 터지는 정체불명의 장애 유발.

---

## 2. 왜 고유 이미지 태깅(Short SHA)이 필요한가?
- **가변 포인터의 신뢰성 한계**
  - `latest` 문자열은 실시간으로 내용이 바뀌는 가변 지시자이므로, 배포 시점에 정확한 이미지 상태(Immutable State)를 명확히 고유 정의할 수 없음.
- **배포 멱등성과 일관성 보장**
  - 클러스터의 모든 노드가 0.1초의 예외 없이 완전하게 동일한 버전의 이미지를 내려받아 구동할 수 있도록 강제해야 함.

---

## 이미지의 불변성 (Immutability)
- **불변 스냅샷 보존**
  - 한 번 빌드되어 레지스트리에 등록된 이미지는 절대로 덮어써선 안 됨.
  - 모든 빌드는 고유한 시리얼 넘버 태그를 지니고 보존되어야 하며, 변경 사항이 있으면 반드시 새 태그 번호를 달아 새 이미지를 릴리즈해야 함.

### 3. 이것은 무엇인가? 고유 이미지 태깅
- **정의**
  - 빌드할 때마다 소스코드의 7자리 Git 커밋 해시(Short SHA)를 이미지 태그로 명명하여, 깃 소스 이력과 도커 이미지 버전을 1대1로 일치시켜 보존하는 관리 기법.
  - 이를 통해 어떤 이미지 태그가 어떤 Git 커밋 소스코드로 빌드되었는지 완벽하게 검증 가능한 가시성을 획득함.

---

## Git Short SHA의 강점
- **절대 겹치지 않는 고유 지문**
  - Git 커밋의 Short SHA는 소스코드가 단 한 글자라도 달라지면 해시 알고리즘에 의해 완전히 다른 문자열로 변환됨.
  - 겹치지 않는 유일성이 보증되므로, 배포 명세서 상에 해당 문자열을 적어주는 것만으로 인프라의 상태 변화를 정확하게 일으킬 수 있음.

---

## latest 방식 vs Short SHA 방식 비교

```
[latest 덮어쓰기 방식 - 장애 시 복구 불가]
─────────────────────────────────────────────────────────────
Registry : ┌──────────────────────────────────────┐ (덮어씀)
           │ [latest] Binary v1.0 ➡️ Binary v2.0  │ 
           └──────────────────────────────────────┘
Cluster  : 노드A(구형 캐시 가동) vs 노드B(신형 가동) ➡️ 소스 파편화 발생
           ※ 에러 인지 후 롤백 요청 ➡️ 구형 v1.0 본체가 소실되어 롤백 실패


[Short SHA 고유 태깅 방식 - 복제 및 롤백 성공]
─────────────────────────────────────────────────────────────
Registry : ┌──────────────────┐  ┌──────────────────┐
           │ [tag: a1b2c3d]   │  │ [tag: 9f8e7d6]   │ (스냅샷 개별 보존)
           │ Binary v1.0      │  │ Binary v2.0      │
           └──────────────────┘  └──────────────────┘
Cluster  : values.yaml 내 tag: "9f8e7d6" 커밋 푸시 ➡️ 전 노드 신형 일제 배포
           ※ 에러 인지 후 롤백 요청 ➡️ tag: "a1b2c3d"로 복원 커밋 ➡️ 즉각 복구 성공
```

---

## 가장 완벽한 불변 포인터: Image Digest
- **Image Digest (SHA256 체크섬)**
  - 태그 문자열 조차도 간헐적 덮어쓰기가 일어날 수 있으므로, 이미지 파일 내용 전체를 해싱한 고유 해시값(`sha256:...`)을 다이렉트로 바라보게 하는 설정.
  - 태그명이 해킹되거나 꼬여도 바이너리 내용물 자체가 다르면 배포되지 않는 궁극의 보안 및 불변성을 제공함.

### 고유 이미지 태깅의 장점
- **인프라 배포의 투명한 감사(Audit)**
  - `git log` 조회만으로 특정 서비스에 어떤 커밋의 소스코드가 가동 중인지 100% 명확히 추적 가능함.
- **Argo CD 변경 감지 최적화**
  - 배포 파일 내부의 태그 텍스트 값이 바뀜에 따라 Argo CD가 변경 사항을 딜레이 없이 인지하고 즉각적인 롤링 업데이트에 돌입함.

---

## 고유 이미지 태깅의 단점 및 극복
- **수동 반영의 번거로움**
  - 빌드할 때마다 튀어나오는 SHA 해시값을 직접 확인하고 수동으로 YAML 파일을 고쳐 커밋해야 하는 귀찮음이 있음.
  - ➡️ **극복 방안**: 실무에서는 CI 도구(GitHub Actions)가 빌드 성공 시 배포 저장소의 YAML 태그 번호를 정규식으로 직접 자동 수정해 커밋하는 자동 업데이트 파이프라인(GitOps Auto-Update)을 연동해 해결함 (기본 흐름 이해를 위해 본 실습은 수동 수정으로 진행).

## 5. 실습: values.yaml 버전 매핑 분석
- **실무형 Helm `values.yaml` 파일 내 이미지 선언부 예시**

```yaml
# day3/k8s/helm/values.yaml
replicaCount: 3

image:
  repository: ghcr.io/social-archive/todo-app
  # latest를 쓰지 않고, GitHub Actions 빌드가 완료된 7자리 Short SHA 입력
  tag: "a1b2c3d"
  pullPolicy: IfNotPresent
```

## 실습: GitHub Actions 빌드 로그에서 SHA 확인
- **태그 번호 획득 방법**
  - 1단계: GitHub 저장소 ➡️ `Actions` 탭 이동.
  - 2단계: 최근 성공한 CI 빌드 Job 클릭 ➡️ 상세 로그의 `Build and Push Docker Image` 단계 확인.
  - 3단계: 출력된 이미지 태그 명칭 뒤의 7자리 깃 커밋 해시값(예: `a1b2c3d`)을 드래그 복사.

## 실습: values.yaml 이미지 태그 갱신
- **Antigravity IDE 또는 텍스트 에디터에서 수정**
  - 복제한 `values.yaml` 파일의 `image.tag` 속성에 복사한 Short SHA 값을 붙여넣고 파일을 저장함:
    ```yaml
    image:
      tag: "a1b2c3d"  # 신규 short SHA값으로 갱신
    ```

## 실습: 이미지 태그 변경사항 Git Push
- **PowerShell에서 실행할 소스 형상 갱신 명령어**

```powershell
# 1. 수정한 values.yaml 파일 변경사항 스테이징 등록

# 3. 리포지토리에 푸시하여 원격 Git에 배포 이력 영구 기록

```

## 💡 강사 팁: 실무 imagePullPolicy 가이드
- **`IfNotPresent` vs `Always`**
  - **개발 환경 (`Always` 권장)**: 개발 시에는 동일한 태그(`latest` 또는 동일 브랜치명)로 이미지를 자주 덮어쓰므로, 컨테이너가 뜰 때마다 무조건 레지스트리에서 새로 받아오게 강제하는 것이 편함.
  - **운영 환경 (`IfNotPresent` 권장)**: 매번 다운로드하면 네트워크 대역폭 낭비가 극심함. 고유한 Short SHA 태그를 쓰고 `IfNotPresent`를 매핑하면, 태그 변경 시에만 노드가 신규 이미지를 다운하므로 속도와 대역폭 면에서 최적의 안정성을 확보할 수 있음.
