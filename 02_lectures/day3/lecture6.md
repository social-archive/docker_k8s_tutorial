## 💡 쉽게 이해하는 비유 (Analogy)
- **현장 땜질 복구 vs 설계도 기반 재건축**
  - **수동 땜질 복구**: 건물에 균열이 가고 무너지려 할 때, 붕괴 현장 속으로 들어가 시멘트를 바르고 지지대를 덧대는 것. 임시방편일 뿐 파드가 다시 켜지면 원래의 부서진 상태로 복구되어 다시 무너집니다.
  - **GitOps 기반 롤백**: 부서진 건물을 만지지 않고, 안전함이 증명된 어제 설계도(Git Revert Commit)를 꺼내어 보관함에 다시 꽂는 것. 건설 로봇(Argo CD)이 알아서 망가진 건물을 밀고 새 건물(정상 파드)을 완벽히 재건축해 줍니다.

---

## 1. 수동 복구의 문제점 (1) 2차 장애
- **임시방편 수정의 데이터 소멸 및 장애 재발**
  - 서비스 마비 시 다급해진 관리자가 서버 노드에 직접 침투해 `kubectl edit` 등으로 환경변수나 마운트 디렉토리 설정 파일을 수동 고쳐 씀.
  - 이 긴급 패치는 파드가 자원 부족(OOM)이나 노드 다운으로 인해 다시 꺼져다 켜지는 순간, 수동 수정 내용이 싹 날아가며 원래의 에러 상태로 원복되어 장애가 재발함.

---

## 1. 수동 복구의 문제점 (2) 이력 유실
- **수동 핫픽스로 인한 이력 꼬임 및 협업 공백**
  - 급한 복구 도중 서버에서 무엇을 고쳤는지 Git 설계 장부나 문서에 기록되지 않음.
  - 다음 날 출근한 교대 엔지니어들은 서버 세부 상태를 인지하지 못한 채 신규 버전을 배포하고, 어제 땜질한 패치가 덮어씌워지며 2차 장애가 재발하고 협업이 붕괴됨.

---

## 2. 왜 GitOps 기반 선언적 롤백인가?
- **인프라 변경 추적성(Traceability)의 사수**
  - 복구 단계에서도 "모든 것은 코드로 기록되고 동기화된다"는 멱등성 규칙이 준수되어야 함.
  - 복구를 위해 쏜 원복 행위 역시 Git 커밋 로그 상에 영구 박제되어, 팀원 모두가 동일한 인프라 타임라인을 공유해야 함.

### 3. 이것은 무엇인가? GitOps 기반 롤백
- **정의**
  - 클러스터의 구동 상태를 직접 편집하지 않고, 인프라 설계서(Git)를 정상 작동했던 이전 특정 커밋 상태로 되돌림으로써 Argo CD가 클러스터를 정상 상태로 자율 동기화하도록 복원하는 방법.

---

## 롤백의 Git 조작: Reset vs Revert
- **Git Reset (역사 말소)**
  - 과거 커밋 시점으로 완전히 시계를 돌리고, 그 이후의 커밋 기록을 이력에서 아예 지워버림.
  - 협업하는 팀원들의 로컬 저장소 브랜치 기록과 엉키므로 GitOps에서는 절대 금기시됨.
- **Git Revert (실수 만회용 덮어쓰기)**
  - 과거 커밋 상태를 복원하는 **새로운 변경사항 커밋**을 이력 맨 상단에 추가함.
  - 기존 에러 커밋 기록도 보존되면서 안전하게 최신 형상을 유지해 Git 충돌을 예방하므로 실무 표준으로 사용됨.

---

## Argo CD UI 롤백과 Auto-Sync Lock
- **Auto-Sync Lock (동기화 충돌 방지 장치)**
  - Argo CD 웹 UI에서 특정 과거 리비전을 선택해 `Rollback` 버튼을 클릭해 강제 복원했을 때.
  - 만약 Git 리포지토리의 최신 코드가 여전히 에러 상태라면, Argo CD의 자동 동기화 기능이 켜져 있을 시 Git 상태로 즉각 롤링 업데이트 덮어쓰기가 되어 롤백이 풀려버림.
  - 이를 막기 위해 UI 롤백 시 Argo CD는 **자동 동기화(Auto-Sync)를 강제로 꺼서 잠금(Lock) 상태**로 전환함.

---

## Argo CD UI Rollback 시의 자동 동기화 잠금 구조

```
[운영자 긴급 UI 롤백 실행]
   Argo CD UI 접속 ➡️ 과거 정상 Revision 2 선택 ➡️ Rollback 클릭
                           │
                           ▼
   [Auto-Sync 자동 잠금 (Enabled ➡️ Disabled)] ➡️ 클러스터 임시 복구 완료
                           │
  ※ 주의: 이 상태에서는 개발자가 Git에 새 패치를 올려도 클러스터에 반영 안 됨 (Sync Lock)
                           │
                           ▼
   [사후 조치 프로토콜 가동]
   개발자가 Git 소스코드를 Revert 커밋 처리해 Push ➡️ Git 상태도 정상화 완료
                           │
                           ▼
   [잠금 해제 (Sync Lock 해제)]
   Argo CD UI ➡️ Enable Auto-Sync 클릭 ➡️ 자율 Reconcile 감시 정상 복원
```

## GitOps 기반 롤백의 장점
- **2차 인적 장애 예방**
  - 사람이 수동 편집 중 오타나 설정을 누락시키는 실수를 원천 방어함.
  - 검증 완료된 100% 안전한 과거의 YAML 설정 규격 그대로 클러스터를 오차 없이 복원함.
- **배포 감사 이력 확보**
  - 복구 흔적까지 Git Commit Log에 정밀하게 기록되어 향후 사후 분석(Postmortem) 작성 시 소중한 증적이 됨.

---

## GitOps 기반 롤백의 단점
- **네트워크 폴링 주기에 따른 롤백 딜레이**
  - Git 커밋, 푸시 후 Argo CD가 이를 감지해 다운하고 배포를 완료하는 데 최대 수십 초의 지연 시간이 소요됨.
  - 1초가 급한 크리티컬한 상용 장애 상황에서는 이 딜레이가 비즈니스에 치명적일 수 있음.

---

## 초긴급 상황에서의 복구 2단계 프로토콜
- **1단계: 선제적 UI/CLI 롤백 (초동 조치)**
  - Argo CD 콘솔에서 과거 정상 리비전으로 강제 롤백을 먹여 1초 만에 서비스를 임시 부활시킴 (동시에 Auto-Sync 잠금 자동 활성화).
- **2단계: Git Revert 및 Lock 해제 (사후 정비)**
  - 서비스가 임시 부활해 숨통이 트인 사이, Git 소스코드를 Revert 커밋하여 Push하고, Argo CD의 Auto-Sync를 다시 켜서(Lock 해제) 정상적인 자율 GitOps 궤도로 인프라를 복원함.

## 5. 실습: 배포 장애 상황 재현
- **오류 상황 유도**
  - 수강생의 `app-deployment.yml` 또는 Helm `values.yaml` 내의 이미지 태그 명칭을 존재하지 않는 오타 태그(예: `tag: a1b2c3d-bug`)로 고의 수정하여 Git Push 수행.

### 실습: K8s 상태 감시로 장애 판정
- **PowerShell에서 실행할 장애 모니터링 명령어**

```powershell
# 이미지 태그 오류로 인해 파드가 켜지지 못하고 에러 상태에 고립되는 상황 추적
kubectl get pods -n todo-app
```
- **체크포인트**: 파드의 STATUS 항목에 **`ImagePullBackOff`** 또는 **`ErrImagePull`**이 출력되는지 확인.

## 실습: A안 - Git Revert 선언형 복원
- **PowerShell에서 실행할 로컬 Git 되돌리기 명령어**

```powershell
# 1. 방금 Push한 오류 커밋의 변경 내역을 정반대로 뒤집는 복구 커밋을 이력 상단에 생성
# (--no-edit 은 에디터 창을 띄우지 않고 커밋 메시지를 자동 작성하는 옵션입니다)
git revert HEAD --no-edit
```

---

## 실습: Revert 소스 푸시 및 Argo CD 동기화
- **PowerShell에서 실행할 동기화 복구 명령어**

```powershell
# 1. 생성된 Revert 복구 설계도를 원격 저장소에 업로드
git push origin main

# 2. Argo CD에 즉각 동기화(Sync) 명령을 전달해 이전의 안전했던 이미지로 파드 재건축 실행
# (Argo CD CLI 또는 웹 화면의 'Sync' 버튼 클릭으로 대체 가능합니다)
argocd app sync todo-app
```

---

## 실습: B안 - Argo CD 임시 긴급 롤백
- **PowerShell에서 실행할 초긴급 롤백 명령어**

```powershell
# 1. Argo CD가 저장하고 있는 과거 배포 성공 리비전 번호 대장 조회
argocd app history todo-app

# 2. 확인된 가장 최근의 정상 릴리스 리비전 번호(예: 3)를 지정해 즉각 롤백 단행
# (이 즉시 Auto-Sync가 차단되며 클러스터 상태가 고정 보존됩니다)
argocd app rollback todo-app 3
```

---

## 실습: [⚠️필수] Auto-Sync 잠금 해제
- **잠금 해제 조치**
  - UI 롤백 완료 후, 반드시 Git 저장소 소스도 Revert하여 상태를 일치시킴.
  - 이후 Argo CD UI ➡️ Application ➡️ `APP DETAILS` ➡️ `ENABLE AUTO-SYNC` 버튼을 클릭하여 잠금을 다시 풀어주어야 함.
  - 해제하지 않으면 향후 신규 릴리스 배포 시 Argo CD가 인프라 동기화를 영구 스킵하는 대기 상태에 빠짐.

## 💡 강사 팁: 운영 배포 전 필수 자가 점검표
- **RCA Checklist (배포 대참사 방어)**
  - [ ] **자원 한도 명시**: `--memory` 혹은 `resources.limits`를 적어 노드 다운 보호장치를 켰는가?
  - [ ] **Readiness Probe 탑재**: 부팅 완료 전 트래픽 유입을 막아 무중단 롤링을 보장했는가?
  - [ ] **Secret 외부 격리**: 소스코드에 중요 패스워드를 박아 깃허브에 밀어 넣지 않았는가?
  - [ ] **고유 태그 사용**: `latest` 포인터를 쓰지 않고 Git short SHA를 고유 낙인으로 찍었는가?
