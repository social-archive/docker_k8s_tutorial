## 오늘 배울 내용
- **주제**: Kubernetes의 선언적 API 모델, 조정 루프(Reconciliation Loop) 및 네임스페이스(Namespace) 개념 이해
- **목표**:
  - 명령형 인프라 조작과 선언형 인프라 선언의 차이점 파악
  - 조정 루프의 3단계(관찰-비교-동작) 메커니즘 습득
  - 네임스페이스를 활용한 논리적 클러스터 자원 격리
  - `kubectl` 명령어와 `k9s` 모니터링 툴을 이용한 클러스터 관찰

## 💡 쉽게 이해하는 비유 (Analogy)
- **선풍기 수동 버튼 vs 에어컨 자동 온도 조절기**
  - **수동 버튼 (Docker CLI)**: 사람이 온도 변화를 감시하다 직접 버튼을 누름. 선풍기가 과열로 고장 나면 사람이 직접 복구해야 함.
  - **에어컨 온도 조절기 (K8s API)**: "희망 온도 24도"로 선언해 두면 조율 센서(조정 루프)가 온도를 상시 대조하여 팬 세기를 조절함. 창문이 열려 온도가 치솟아도 센서가 자동으로 24도가 될 때까지 냉방을 계속 가동해 사람이 신경 쓸 필요가 없습니다.

## 1. 기존 도커 환경의 문제점 (1) 명령형 인프라
- **명령형 제어 (Imperative Control)의 한계**
  - 웹 컨테이너 5개를 늘리려면 사람이 직접 `docker run` 명령어를 포트를 다르게 지정해가며 5번 쳐야 함.
  - 실행 중간에 디스크 용량이 꽉 차거나 네트워크 단절로 4번째 실행이 실패하면, 인프라는 의도한 5개가 아닌 3개만 올라간 애매한 상태로 방치됨.
  - 이처럼 수동 명령에 의존하면 설계한 형상과 실제 인프라 상태가 다르게 변하는 **상태 표류(Configuration Drift)** 현상이 빈번히 발생함.

## 1. 기존 도커 환경의 문제점 (2) 자가 치유 부재
- **자가 치유 기능의 한계**
  - 백그라운드로 실행 중이던 단일 도커 컨테이너가 내부 메모리 릭으로 죽어버렸을 때.
  - 도커 데몬은 컨테이너가 멈췄음을 인지해도 스스로 이를 되살려내지 않음.
  - 운영자가 쉘 스크립트나 크론탭을 활용해 주기적으로 헬스를 체크하고 `restart` 명령어를 던지도록 땜질식 감시 코드를 직접 작성해야 하는 애로사항 발생.

## 2. 왜 Kubernetes 기본 구조가 필요한가?
- **단발성 명령어의 한계 극복**
  - 도커 엔진은 프로세스를 키고 끄는 조작기일 뿐, 인프라의 전체 형상(아키텍처)을 지속적으로 계측하고 상태를 유지하는 능력이 없음.
- **인프라 자율화의 실현**
  - 지속적인 감시와 대조를 통해 고장 난 부품을 자동 교체하고, 인프라를 항상 일정한 안전 구역 내에서 관리하기 위함.

## K8s 핵심 사상 (1) 선언형 API
- **Declarative API**
  - "어떻게(How) 구축할 것인가"를 지시하지 않음.
  - "도달해야 할 최종 인프라 상태(What)"를 YAML 설정 문서로 명세하여 제출함.
  - 제출된 YAML 선언문은 쿠버네티스 데이터베이스(`etcd`)에 단일 진실 공급원(SSOT)으로 영구 등록되어 관리됨.

## K8s 핵심 사상 (2) 지속적 조정 루프
- **Reconciliation Loop**
  - 쿠버네티스의 심장에 해당하는 지속적인 피드백 조율 엔진.
  - 데이터베이스에 기록된 **목표 상태(Desired State)**와 물리 서버에 배포되어 작동하는 **현재 상태(Actual State)**를 상시 계측하여 대조함.
  - 양측에 차이가 발견되는 즉시 스스로 복구 명령을 작동시켜 격차를 좁힘.

## 3. 이것은 무엇인가? Kubernetes 기본 구조
- **정의**
  - 선언된 최종 목적지 상태를 향해 끊임없이 동기화하는 자율 인프라 제어 플랫폼.
  - 사용자는 설계 도면(YAML)만 제공하면 되며, 서버 다운 시 부활시키는 작업, 트래픽을 늘리는 작업 등은 쿠버네티스 백그라운드 스레드가 알아서 처리함.

## 명령형(Imperative) vs 선언형(Declarative) 비교

| 항목 | 명령형 (Imperative) | 선언형 (Declarative) |
| :--- | :--- | :--- |
| **제어 관점** | "우회전하고 100m 가서 멈춰라" (과정 기술) | "서울역으로 가라" (최종 목적지 지정) |
| **도구 예시** | Docker CLI 명령어 반복 실행 | Kubernetes YAML 매니페스트 배포 |
| **실패 대처** | 중간 단계 실패 시 시스템 엉킴 | 목표에 도달할 때까지 자동 시도 및 조율 |
| **멱등성** | 동일 명령어 중복 시 에러 우려 | 동일 매니페스트 반복 적용 시 중복 처리 없음 |

## 조정 루프(Reconciliation Loop)의 3단계
- **1단계: 관찰 (Observe)**
  - 실제 워커 노드들에서 가동 중인 컨테이너 리소스를 실시간 계측함 (현재 상태: 2개 구동).
- **2단계: 비교 (Diff)**
  - 목표 상태(목표: 3개 구동)와 현재 관찰 상태(2개)를 비교해 1개 부족함을 인지함.
- **3단계: 행동 (Act)**
  - 부족함을 해결하기 위해 신규 컨테이너 1개를 새로 띄우는 API를 호출해 동기화함.

        RC --> |"5-2. 비교 (목표 3 vs 실제 2 = 1개 부족)"| Compare{"차이 분석"}

## 논리적 장벽: 네임스페이스 (Namespace)
- **네임스페이스 (Namespace)**
  - 하나의 물리적 쿠버네티스 클러스터 자원을 여러 개의 논리적 작업 구역으로 쪼개서 사용하는 가벽 격리 장치.
  - 부서별, 프로젝트별, 환경별(dev, staging, prod)로 자원이 섞이지 않도록 방어막을 쳐서 관리의 위험성을 제한함.

## Kubernetes 기본 구조의 장점
- **자율 운영 인프라의 완성**
  - 새벽에 물리 서버 장비가 죽어도, 조정 루프가 몇 초 만에 다른 서버에 대체 컨테이너를 살려내어 자가 복구하므로 장애 복구 피로도가 격감함.
- **배포 신뢰성(멱등성)**
  - 텍스트 파일(YAML)에 적힌 명세서가 인프라 최종 형상을 보장하므로, 환경 차이로 인한 수동 배포 실수가 근본적으로 차단됨.

## Kubernetes 운영의 한계: 내부 가시성 상실
- **관제 피로도와 블랙박스화**
  - 쿠버네티스는 스스로 파드를 지우고, 이주시키고, IP를 바꾸는 등 끊임없이 자율적으로 상태를 교정함.
  - 이로 인해 CLI 상에서 단순 `kubectl get pods` 명령어만 단발성으로 쳐서는 실시간 상태 변동 및 에러 로그의 흐름을 파악하기 매우 어려움.

## 내부 가시성 확보: k9s 대시보드
- **k9s**
  - 터미널 기반의 고성능 실시간 모니터링 대시보드 도구.
  - 별도의 헤비한 웹 UI 없이 가벼운 터미널 커맨드 환경에서 모든 쿠버네티스 리소스의 기동 상황, 에러 이벤트, 실시간 로그 스트림을 GUI 형태로 추적할 수 있어 가시성 결여 문제를 깨끗이 해결함.

## 5. 실습: 네임스페이스 선언서 작성
- **실무형 `namespace.yaml` 선언서 예시**

```yaml
apiVersion: v1
kind: Namespace
metadata:
  name: todo-app
  labels:
    env: development
    project: todo-service
```
- **체크포인트**: `apiVersion`, `kind`, `metadata`로 이어지는 쿠버네티스 매니페스트 기본 스키마 확인.

## 실습: 선언형 배포 적용 및 조회
- **PowerShell에서 실행할 네임스페이스 배포 및 확인 명령어**

```powershell
# 1. 작성한 선언서(YAML)를 적용하여 클러스터 상태 업데이트
# (동일 명세서 반복 적용 시 중복 생성 없이 기존 자원을 유지합니다)
kubectl apply -f namespace.yaml

# 2. 현재 클러스터에 활성화된 전체 네임스페이스 목록 조회
kubectl get namespace
```

## 실습: 네임스페이스 내부 리소스 전체 조회
- **PowerShell에서 실행할 리소스 탐색 명령어**

```powershell
# 특정 네임스페이스(todo-app) 내부에 존재하는 모든 API 리소스(Pod, Service 등) 일괄 조회
# (-n 옵션은 namespace 필터를 의미합니다)
kubectl get all -n todo-app
```

## 실습: k9s 대시보드 기동
- **PowerShell에서 실행할 실시간 감시 도구 기동**

```powershell
# k9s 터미널 대시보드 실행
# 키보드 방향키와 단축키(마우스 클릭 지원)로 리소스 탐색 가능
k9s
```
- **주요 단축키**:
  - `0`: 전체 네임스페이스 보기
  - `:namespace`: 네임스페이스 목록으로 이동
